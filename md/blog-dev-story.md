# 동기

어린 시절 스타크래프트 맵 에디터를 시작으로, 지금까지 아마추어 개발자로써 살아온 인생을 회상 해보면, 굉장히 많은 시도 들을 했던 것 같습니다. 비록 과한 호기심에서 발생한 의미 없는 행위들이 대부분 이었으나, 그런 경험들이 하나 하나 쌓여서 지금의 저를 만들었다는 생각이 듭니다. 그러나 그런 행위들에 대한 체계적인 기록 부족으로 많은 아쉬움이 남습니다. 많은 경험들이 기억 속에서 희미해지거나 사라졌고, "지금 이 순간도 기록하지 않으면 미래에는 잊혀지겠다." 라는 생각이 들었습니다. 앞으로 어떤 행위를 하든 블로그에 체계적인 기록을 남겨서 같은 실수를 반복하지 않으려고 합니다.

# 정적 블로그를 선택한 이유

tistory 블로그나 네이버 블로그 등의 블로그 서비스들을 택하지 않은 이유는 간단합니다. 제가 전달하고 싶은 내용에 비해 배꼽이 너무 컸습니다. 전달하고 싶은 글의 용량보다 곁다리로 딸려오는 온갖 소스코드들이 더 많은 상황이 불편했습니다. 쓸 줄도 모르고, 사실 쓸 필요도 없는 온갖 고급 관리 기능들의 존재가 마음에 들지 않았습니다. 정적 블로그는 말 그대로 static file 들을 제공 해주는 형태이기 때문에 정말 필요한 정보만 사용자한테 보여 줄 수 있었고, 그 점이 마음에 들었습니다. 지금 글을 쓰고 있는 시점에서는 github.io에 블로그를 통째로 올려놔서 비용 문제도 없는 상태지만, 추후에 블로그 서버를 직접 운영할 때 서버 운영비가 크게 들지 않을 것 같아서 비용 적인 측면도 좋았습니다. 

# 정적 블로그 생성기도 만들게 된 이유

처음에는 유명한 정적 사이트 생성기들을 사용하려고 했습니다. 그러나 Jekyll를 사용하려면 Ruby 언어를 같이 배워야 해서 익숙한 언어로 만들어 진 Hugo를 알아봤었습니다. Hugo는 많은 부분을 Go가 제공 해주는 template에 의존하고 있었는데, "나도 Go 좀 해봤는데 이 정도는 직접 만들 수 있겠다."라는 근거 없는 자신감이 들어서 Go로 정적 블로그 생성기를 만들게 되었습니다.

# 개발 과정: 정적 블로그 생성기

소스코드: [github.com/wirekang/blogen](github.com/wirekang/blogen)

이름은 blogen 이라고 짖게 되었습니다. blog + generator인데, 프로젝트 디렉터리를 생성할 때 대충 지은 임시 이름 이었다가 계속 사용하게 되었습니다.

## 목표

역시 마크 다운 만큼 적절한 게 없었기에, 다음과 같은 시나리오로 작동하는 프로그램을 계획했습니다.

1. 프로그램 실행
2. 작성된 파일들로부터 메타 데이터(제목, 날짜, 태그)와 마크 다운 부분 분리
3. 메타 데이터들은 메모리에 저장
4. 마크 다운들은 HTML로 파싱하여 따로 디렉터리에 저장(메모리 부족을 고려하여)
5. 저장된 템플릿을 읽어서 정적 사이트 생성

마크 다운 파일을 HTML파일로 파싱하는 역할은 [github.com/gomarkdown/markdown](https://github.com/gomarkdown/markdown) 패키지를 사용했습니다. Go의 생태계가 좁아서 마크 다운 파싱을 직접 구현해야 될 줄 알았는데, 의외로 패키지가 있었습니다. 나중에 규모가 커질 것을 대비하여 파싱 결과를 메모리에 저장하지는 않았습니다. md5로 캐싱 해두는 기능도 구현해서 최대한 빠른 작동을 노렸습니다.

## 난관

패키지 구조에 대한 고찰 때문에 코드를 쓰고 지우기를 반복했습니다. 깔끔한 프로그램을 만들고 싶었던 나머지, Go언어만의 특이한 패키지 개념을 과도하게 의식했습니다. 결국 간단한 프로그램 치고 너무 많은 패키지를 만들었다가 패키지 간의 통신이 원활하지 못하여 다시 줄여서 작성하고..를 반복했습니다. 비즈니스 로직을 완벽하게 분리하고 싶은 욕심 또한 큰 영향을 줬는데, 애초에 비즈니스 로직을 완벽하게 분리할 수 없다는 사실을 받아들이는 것으로 해결하였습니다. 결과적으로 파일, 디렉터리 관련 부분을 담당하는 **fl** 패키지와 생성을 담당하는 **gen** 패키지, 그리고 커맨드 라인을 처리하는 **main** 패키지로 나누었습니다.

# 개발 과정: 정적 블로그

소스 코드: [github.com/wirekang/blog](https://github.com/wirekang/blog)

정적 블로그를 생성해주는 프로그램을 만들었으니, 진짜 블로그를 만들 차례였습니다.

## 목표

하나의 글은 하나의 카테고리 대신에 여러 개의 태그를 달게 했습니다. 벨로그에서 이런 형식을 봤었는데, 굉장히 인상적이었습니다.

### 메인화면

1. 블로그 제목
2. 블로그 설명
3. 태그 목록
4. 모든 글의 제목과 작성 날짜

### 태그를 누르면

1. 블로그 제목
2. 태그 목록
3. < ** 태그가 포함된 글>
4. 해당 태그가 포함된 글의 제목과 작성 날짜

### 글을 누르면

1. 블로그 제목
2. 글 제목과 작성 날짜, 태그들
3. 글 내용
4. 관련된(태그가 겹치는)글 목록

## 프로토타입

위의 목표에 따라서 가장 처음 구현했던 블로그입니다.

 [링크](https://wirekang.github.io/blogen/example/out/index.html)

디자인을 전혀 안 한 상태에서 기본적인 뼈대를 잡았습니다. 이 때만 해도 템플릿 파일 네 개를 다 합쳐도 5~60 줄이 안되었습니다. 원하는 것도 뼈대에서 크게 다르지 않았기 때문에 금방 끝날 줄 알았습니다.

## 난관

css를 하는 것과 디자인을 하는 것은 전혀 다른 일이라는 것을 알았습니다. 저는 css에 대한 이해가 어느 정도 된 상태였기 때문에 금방 일을 마무리 할 줄 알았는데, 아무리 해봐도 **멀쩡해 보이는** 블로그 구현이 힘들었습니다. 지금 보고 계시는 이 블로그의 후진 디자인은 많은 삽질 끝에 만들어 진 결과물 입니다. 디자인, 미적 감각 같은 것과는 먼 사람으로써, 이 과정이 제일 어려웠습니다.

# 개발 과정: 댓글 서비스

소스코드: [github.com/wirekang/comment](https://github.com/wirekang/comment)

원래는 예정에 없었습니다. 그런데 시간도 좀 남고 막상 블로그가 너무 적적하기도 하여 댓글 서비스를 만들기로 하였습니다.

## 목표

1. 이름, 비밀번호, 댓글 내용을 입력하면 javascript가 ajax를 통해 서버로 해당 정보를 전송
2. 서버에서는 해당 정보를 받아서 데이터베이스에 저장, 요청 시 전송

아주 간단한 api서버가 필요했습니다.

## 날먹 실패

처음에는 닷홈 같은 무료로 호스팅을 해 주는 업체에서 공짜로 호스팅을 받아다가 기본으로 설치된 php와 mysql을 통해서 댓글 서비스를 날먹하려고 했습니다. 그런데 이런 행위가 해당 업체 규정 위반이라, 제재를 피하기 위해 어리버리한 위장용 사이트를 만들던 도중 자괴감을 느꼈습니다. ( 아직 남아있습니다. [링크](http://wirekang.dothome.co.kr) ) 안 그래도 미래도 없는 언어인 php를 사용해서 서버를 구성하는 것에 회의감도 들어서, 이 참에 **node**를 한번 써 봐야겠다 싶어 AWS free tier를 서버로 사용하기로 했습니다. 기본적으로 무료이긴 했지만 이미 24 시간 풀로 사용 중인 인스턴스가 하나 있어서 다음 달 부터 과금이 될 예정입니다.

## 난관

구현 자체는 굉장히 별 거 없었습니다. 데이터베이스도 외부 서비스를 따로 돌리지 않는 sqlite3를 사용하여 경량, 소규모 api 서버를 금방 만들었습니다. 그래도 역시 난관이 좀 있었습니다.
